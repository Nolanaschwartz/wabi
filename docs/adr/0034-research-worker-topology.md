# The research worker is a separate, always-on NestJS service that owns its own schedule, config, and run history

`@wabi/research` stops being a one-shot CLI (`ts-node src/run.ts` behind a hand-rolled `loadDotenv()`) and becomes a **separate, always-on NestJS HTTP service** bound to its own port (`:3002`; web owns :3000, bot :3001). It boots a real `ConfigModule` (reading the root `.env`, like the bot), gains `DATABASE_URL`, and is the **single owner of the `research-run` pg-boss queue** — it both schedules the queue and consumes it. It exposes an authenticated `/admin/research/*` HTTP surface (behind the same timing-safe `x-admin-secret` `AdminGuard` the bot uses, reusing `ADMIN_API_SECRET`), and the web app gains an operator-gated `/admin/research` page + `/api/admin/research/*` proxy that forwards to it.

The bot gains **no** new responsibilities. It keeps `/admin/strategies/ingest` (the trust gate → `StrategyDraft` → Qdrant) and learns nothing about scheduling. The worker remains a **pure candidate producer**: it still POSTs candidates over HTTP to the bot's trust gate (via a configurable `STRATEGY_API_URL`, default = the bot, falling back to today's `BOT_BASE_URL`).

## Why

- **An operator needs to control the research run without shell access.** Today there is no schedule, no run-now, no run history, and no admin surface — running the agent requires invoking a CLI, and its topics and bounds are env-only. Making the worker a long-lived service with its own admin HTTP surface is what lets the schedule, topics, and bounds become DB-backed and browser-editable (ADR-0018 named a `research-cron`; this is where it lives).
- **A heavy research run must never share a process with the crisis-safe DM hot path.** The bot is always-on and never serverless (ADR-0019/0020) precisely because the crisis→coaching pipeline can't block. A long, token-burning, network-heavy research run belongs in its **own** process so it can never contend with or destabilise that path. Separate service = separate failure domain.
- **The worker can own a schedule cheaply because it already owns a Postgres connection's worth of state.** Once the worker has `DATABASE_URL` for its own config/topics/runs tables, it can be the single owner of the `research-run` pg-boss queue — scheduling and consuming the very queue it asserts. No second process, no cross-service scheduling protocol.

## Scope and bounds

- **DB access is scoped, not opened.** The worker reads/writes **only** `ResearchConfig`, `ResearchTopic`, `ResearchRun`, and the pg-boss queue. It never reads or writes `User`, any inner-state model, or `StrategyDraft` directly. This **amends ADR-0033**: "the worker keeps no DB credentials" → "the worker has no *user-data* access." The isolation boundary (ADR-0002) holds — the change is from "no DB at all" to "its own tables + its own queue, nothing else."
- **The trust gate stays the only path into the strategy library.** Candidate ingest still flows HTTP→bot (`POST /admin/strategies/ingest`, `GET /admin/strategies/seen`), so all trust/safety/dedup logic stays single-sourced on the bot (ADR-0012/0033). The worker decides *what to read and extract*; the bot decides *what becomes a strategy*.
- **DB is the source of truth for config after first boot.** On first boot the worker idempotently upserts the `ResearchConfig` singleton (Int defaults mirror today's `RESEARCH_MAX_*` env defaults) and seeds `ResearchTopic` from the existing `seed-topics.ts` only when the table is empty. After that, `loadBounds()` / `SEED_TOPICS` / `RESEARCH_MAX_*` demote to **bootstrap seed defaults only**. Repeated restarts never duplicate or reset anything (the singleton upsert uses an empty `update` so operator edits are never clobbered).
- **Forward-compat seam.** `STRATEGY_API_URL` (default = bot) is the ingest target. A future extraction of the Strategy knowledge base into its own service is then an env flip, not a code change (that extraction is a separate ADR, out of scope here).
- **The pure core is wrapped, not rewritten.** `runResearch`, `ResearchAgent`, `agent/*`, `sources/*`, and `BotClient` are untouched; later slices wrap them in a runner service. The retired piece is only the standalone `run.ts main()` + `loadDotenv()` entrypoint, replaced by `main.ts` + `ConfigModule`. The lazy-getter rule still holds — `getProvider()` re-reads `process.env`, which `ConfigModule` populates at bootstrap; no env-derived state is cached in a field.

## Consequences

- The system runs **four** processes locally (`pnpm dev`): web :3000, bot :3001, research :3002, plus web's own server. Research is non-critical by construction — if Postgres or pg-boss is down, scheduling is inert and the admin endpoints surface a service-unavailable error, but **nothing else in the system is affected**.
- The worker gains `DATABASE_URL`, `RESEARCH_PORT` (default 3002), `RESEARCH_TZ` (default `UTC`), `STRATEGY_API_URL` (ingest target, default = bot), and reuses `ADMIN_API_SECRET` to guard its own endpoints. Web gains `RESEARCH_API_URL`. `packages/shared/.env` stays `DATABASE_URL`-only; the worker's `ConfigModule` reads the root `.env`.
- Two new Prisma tables (`ResearchConfig`, `ResearchTopic`) land now; `ResearchRun` follows with the runner slice. They are disjoint from `StrategyDraft`/`ProcessedSource` and owned solely by the worker.
- Slice 01 (this ADR) lands the foundation: the Nest service, the seeded config, the guarded read endpoint, the web proxy + read-only page. Later slices add topic editing, bounds tuning, scheduling, run trigger + history, and the real run.
