# Production tracing adopts the Langfuse OpenTelemetry SDK; crisis safety rides the synchronous crisis gate

Production tracing moves from the hand-rolled `LangfuseIngest` kernel + `TracePayloadBuilder` (ADR-0037) to the official Langfuse v5 OpenTelemetry SDK (`@langfuse/tracing` + `@langfuse/otel`), with scores on `@langfuse/client`. Both always-on services migrate — the bot's `LangfuseTracer` and research's `ResearchTracer` — and the kernel is **deleted**. The Langfuse provider is an **isolated `NodeTracerProvider`** (never the global one), so it never contends with Sentry's global OTEL provider. The AI SDK's automatic generation spans (`experimental_telemetry`) are enabled **only for LLM calls that run strictly *after* the crisis verdict**; everything that runs before it is traced by hand and dropped on crisis, exactly as today. A content-agnostic OTEL bootstrap lives behind a new `@wabi/shared/otel` subpath; crisis policy stays in the bot's Wellbeing context.

This **amends ADR-0037**: the "Langfuse ingestion is a hand-rolled shared kernel" half is superseded. The `generate(role, opts)` LLM kernel and the "fail policy stays per-context" half of ADR-0037 are unchanged.

## Why

- **ADR-0037's anti-OTEL reasoning was predicated on *global* auto-instrumentation.** The objection — "OTEL needs `NodeSDK` booted before app logic, with import-order constraints, and would pull OTEL into `@wabi/shared`/web" — is a property of global module-patching instrumentation. We don't use it. With an **isolated provider** + the AI SDK's explicit `experimental_telemetry.tracer` option (route spans to a non-global `TracerProvider`) + `setLangfuseTracerProvider()` (bind the manual-span helpers to it), nothing is monkey-patched. The only requirement is that the tracer exists before the first below-gate `generate()` — request time, long after boot — so the provider initialises inside Nest *after* `ConfigModule` populates env, sidestepping the lazy-config load-order trap entirely.
- **The SDK earns the switch on three axes the operator asked for:** vendor-maintained ingestion format (stop hand-maintaining the batch-envelope JSON and tracking Langfuse API changes), richer nested span trees (true OTEL parent/child + automatic generation spans with model/usage), and one coherent SDK story with the eval/read plane already adopting `@langfuse/client`.
- **Crisis safety gets *stronger*, not weaker, by leaning on a property that already exists.** The crisis classifier is awaited at the top of the turn; coaching only runs on a non-crisis verdict (`CoachHandler` is documented as only ever invoked on a turn that cleared every gate). So the *only* content-bearing LLM calls that run before the verdict are the ones in the pre-verdict `Promise.all` (`classifier`, `router`, and the retrieval embedding) — and all of those are traced **manually and dropped on crisis**, never auto-instrumented. Every auto-instrumented call runs strictly below the gate, on a turn already known to be safe. There is therefore no window in which crisis content reaches an auto-emitted span.

## Considered options

**Why not the global `NodeSDK` (the "obvious" OTEL setup).** `@sentry/node` v10 is built on OpenTelemetry and registers the global `TracerProvider` on `Sentry.init` (already running at the top of `main.ts`). Making a global Langfuse `NodeSDK` coexist requires `skipOpenTelemetrySetup: true` *plus* manually re-adding Sentry's `SentryContextManager`, `SentryPropagator`, `SentrySampler`, keeping `httpIntegration` for request isolation, and managing ESM loader hooks — fragile wiring bolted onto the working error path. An isolated provider leaves Sentry completely alone. The manual-span ergonomics that a global provider would have given us for free are recovered by `setLangfuseTracerProvider()`, so the global path's only remaining advantage (zero-arg auto-instrumentation of *other* libraries) is something we explicitly don't want — Langfuse is the only backend.

**Why not enable auto-telemetry everywhere + drop crisis at export (`shouldExportSpan`/`mask` as the primary guarantee).** If content-bearing spans are auto-created *before* the verdict, crisis safety depends on the export-stage filter running after the crisis set is populated — a timing race against the `BatchSpanProcessor`'s flush (which fires on a queue-size threshold under burst regardless of the scheduled delay), and `forceFlush()` is global, so one turn's flush drains other concurrent turns' pre-verdict spans. That makes the highest-stakes guarantee in the repo timing-dependent and relocates a safety invariant into infra — exactly what ADR-0037 warned against. Rejected. The buffering-span-processor variant (hold spans per trace id until an explicit turn-end signal) *would* close the race, but it is unnecessary once auto-telemetry is confined below the gate: there is no pre-verdict content span to buffer.

**Why `mask` is dropped, not kept as a second layer.** ADR-0024 deliberately retains *full* non-crisis coaching content for eval data, so a blanket content-scrubbing `mask` fights that intent; and the JS `mask` can only redact, not drop, so it cannot satisfy ADR-0024's "crisis traces dropped *entirely*." The trace-id-keyed `shouldExportSpan` drop covers the crisis case completely (whole tree, root included), so `mask` would add a moving part on the safety path for no gain.

## Consequences

- **The invariant is auditable and ties to one code location:** *no auto-instrumentation above the crisis gate* (`coaching.service.ts`'s `if (classification === 'crisis')` short-circuit). Below-gate `generate()` calls pass `experimental_telemetry: { isEnabled: true, tracer, recordInputs: true, recordOutputs: true }`; above-gate calls pass nothing. A new pre-verdict LLM call that forgot this is the one thing that could regress safety — so the `shouldExportSpan` backstop and a test guarding the boundary exist for exactly that.
- **Crisis backstop = `shouldExportSpan` only, keyed on trace id, dev-aware.** A bounded process-wide `Set<traceId>` of crisis-latched ids lives in the bot (Wellbeing); the orchestrator latches synchronously the instant the verdict is crisis, before any span of that turn ends. The closure composes with the default filter: `isDefaultExportSpan(otelSpan) && !(isProd && crisisSet.has(traceId))`. Dev (`localFullFidelity`) keeps crisis traces for classifier debugging; prod drops the whole crisis trace. No race: the synchronous latch precedes every span-end, and the batch processor only exports ended spans.
- **`@wabi/shared/otel` is a new content-agnostic subpath** (`@langfuse/tracing`, `@langfuse/otel`, `@opentelemetry/sdk-node`), isolated like ADR-0037's `ai`/`@ai-sdk/openai` and **not** re-exported from the barrel, so `@wabi/web`'s bundle never pulls OTEL. It exports `createLangfuseTracing({ sampleRate, shouldExportSpan? }) → { tracer, shutdown }`; the bot supplies the crisis-aware `shouldExportSpan`, research supplies none.
- **The OTEL trace id becomes the canonical turn id** (`crypto.randomUUID()` is removed); `withTraceId` log correlation and scores key on it. Sampling is `TraceIdRatioBasedSampler` resolved once at boot. Scores move to `@langfuse/client.score.create` (full-fidelity, independent of span sampling); the crisis-score branch and the content-free trace upsert are removed (production scores are emitted below the gate, hence inherently non-crisis).
- **Span vocabulary is preserved.** `turn` (root span), `classify`/`intent` (manual generations, above gate), `retrieval` (manual span, above gate), `memory` (manual span, below gate), `coach` (a thin manual `coach` parent below the gate with the AI-SDK auto generation nested under it, so name-based eval queries stay stable).
- **`LangfuseIngest` + `TracePayloadBuilder` are deleted**, their tests with them; research's `ResearchTracer` re-expresses its run/paper tree (`gate`/`extract`/`dedup`) over the same OTEL setup with no gate (no crisis concept). One tracing mechanism across both always-on services.
- **Tracing still never breaks the hot path (ADR-0021):** provider init is fail-open (missing `LANGFUSE_*` or any init error → no tracing, service boots), and flush/shutdown runs on `OnApplicationShutdown`.
